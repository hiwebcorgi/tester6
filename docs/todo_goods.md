# [DEV-RULE] ZeroPack 제품 개발 원칙

1. [ ] 디자인은 '코드'로 고정하고, 내용은 '데이터(JSON/MD)'로 분리했는가?
2. [ ] 사장님이 사진을 아무리 큰 걸 올려도 사이트가 느려지지 않는가?
3. [ ] 업종 전환 시 코드 수정 없이 JSON 파일 하나로 레이아웃이 바뀌는가?
4. [ ] 배포 직후 별도 설정 없이 네이버/구글에 등록될 준비가 되었는가?

---

1. 마스터 템플릿의 '모듈화' (Modular Architecture)
10개의 템플릿을 각각 따로 만들면 유지보수가 지옥이 됩니다. 하나의 **'강력한 엔진(Core)'**을 만들고 업종별로 '부품(Section)'만 갈아 끼우는 구조여야 합니다.

컴포넌트화: 히어로 섹션 A타입(이미지형), B타입(슬라이드형) 등을 미리 다 만들어두고, JSON 설정 파일 하나로 끄고 켤 수 있게 설계하세요.

디자인 시스템: 폰트, 컬러, 버튼 둥글기(Border-radius) 등을 변수(Variable)화하여, 사장님이 브랜드 컬러 하나만 바꿔도 사이트 전체 분위기가 바뀌게 해야 합니다.

2. 사장님을 위한 '방어적 GUI' 설계 (Safety Design)
사장님이 수정을 하다 디자인을 망가뜨리는 것이 가장 큰 리스크입니다. 이걸 기술적으로 막아야 AS가 0이 됩니다.

글자 수 제한: 제목 영역에 소설을 쓰지 못하도록 텍스트 입력창에 글자 수 제한(Max Length)을 거세요.

이미지 최적화 자동화: 사장님이 10MB짜리 사진을 올려도 Next.js의 <Image /> 태그가 자동으로 리사이징하고 WebP로 변환하게 세팅해야 합니다. (이래야 속도가 유지됩니다.)

수정 범위 고립: 레이아웃(틀)은 코드에 고정하고, 오직 '내용(텍스트, 이미지)'만 TinaCMS에서 건드릴 수 있게 영역을 분리하세요.

3. '배포 버튼' 하나로 끝나는 구조 (One-Click Deployment)
유저님이 고객의 정보를 받았을 때, 코드를 수정하고 터미널을 두드리면 안 됩니다.

GitHub Template: 완성된 코드를 'Template Repository'로 만드세요.

환경 변수(Env) 활용: 사이트 이름, 로고 URL, 사장님 이메일 등을 Vercel의 환경 변수로 관리하면, 코드 한 줄 안 고치고 설정값만 입력해서 사이트를 생성할 수 있습니다.

도메인 연결 자동화: Vercel API를 이용해 도메인 연결까지 자동화할 준비가 되어 있는지 체크하세요.

4. '깡통 사이트'가 되지 않기 위한 기본 세팅 (The Basics)
유지비 0원만큼 중요한 것이 **'검색 결과'**입니다.

SEO 프리셋: 모든 페이지에 기본적으로 Canonical URL, Open Graph(공유 시 미리보기), Robot.txt가 자동으로 생성되도록 초기 설정을 완벽히 해두세요.

접근성 및 성능: Google Lighthouse 점수가 최소 90점 이상 나오는지 확인하세요. "유지비도 없는데 속도까지 미쳤다"는 소리가 여기서 나옵니다.
